    Week07 WIL #

    1 - CPU 스케쥴링을 알아야 하는 이유는 컴퓨터에게 일을 더 잘 시키기 위해서 입니다. 그렇다면 일을 잘한다는 것은 무엇일까요? 그 평가 기준에는 반환 시간, 공정성, 응답시간이 있습니다. 반환 시간은 작업 시작 시간 에서 끝낸 시간을 차인 즉 실제 작업 한 시간이고 공정성은 작업들이 공평하게 자원을 쓸 수 있는가 하는 것이고 응답 시간이란 요청 받은 시간으로 부터 처음 실행될 때 까지의 시간 차를 의미 합니다.

    2 - 오늘 다룰 CPU 스케쥴링의 종류는 다음과 같습니다.

    3 - 현재 스케쥴링의 문제점은 무엇일까요? 사진에 보이다시피 들어온 순서대로 작업이 진행됩니다.

    4 - 이런 경우 앞서 들어온 작업 A의 길이가 굉장히 긴 경우 뒤에 들어온 B, C 의 작업들은 A가 종료되기를 기다려야합니다. 작업의 길이가 크게 다른 경우 반환 시간의 평균은 크게 늘어나고 이 현상을 convoy effect 라고 합니다.

    5 - 작업의 길이가 문제라면 짧은 작업 먼저 처리하는 방식은 어떨까요? 만약 작업이 모두 동시에 들어온다고 하면 SJF가 최적의 방법이 될 수 있습니다. 하지만 현실에서 모든 작업을 동시에 시킬 순 없겠죠.

    6 - 그림과 같이 긴 A 가 먼저 도착하고 B, C 가 나중에 도착한다면 앞서 본 convoy effect 가 다시 발생하게 됩니다.

    7 - 이번에는 스케쥴러가 하던일을 멈추고 작업을 스위칭 할 수 있다면 현재 실행 중인 작업의 잔여 시간과 새로운 작업의 잔여 시간을 비교하여 가장 작은 작업을 스케쥴링 합니다.

    8 - 만약 평가 기준이 반환 시간 하나라면 STCF는 훌륭한 스케쥴링 정책입니다. 하지만 시분할 컴퓨터의 등장으로 사용자는 터미널을 통해 시스템과 상호작용 할 수 있게 됨에 따라 상호작용을 원활히 하기 위해 응답시간 이라는 새로운 평가기준이 생기게 되었습니다. 앞서 본 SJF나 STCF 같은 경우엔 동일한 길이의 작업이 동시에 3개 스케줄 된다면 3번째의 작업은 응답 시간이 상당히 늦어지게 됩니다.

    9 - 이러한 응답 시간 문제를 해결 하기 위해 라운드 로빈 스케줄링 방법이 고안 되었습니다. 라운드 로빈 방식에서는 타임 슬라이스 길이 만큼 작업을 진행한 후 큐의 다음 작업으로 전환됩니다. 타임 슬라이스 길이가 짧아질 수 록 응답 시간을 줄일 수 있지만 다음 작업으로의 전환이 잦아지므로 문맥 교환 비용을 고려해야 합니다.

    10 - 앞서 2가지 종류의 접근법을 살펴보았습니다. 하나는 남아있는 작업중 제일 짧은 작업을 수행하고 반환 시간을 최소화 하고 나머지 하나는 모든 작업을 번갈아 하고 응답시간을 최소화 하였습니다. 반환시간과 응답시간의 상충관계 외에도 정확한 스케줄링을 위해선 작업 시간을 미리 알기까지 해야합니다.

    11 - 앞서 말했던 세가지의 요구 조건을 고려하는 스케쥴링 방식 중 하나는 MLFQ입니다.MLFQ가 해결하려고 하는 기본적인 문제는 프로세스가 얼마나 큰 작업인지 모른다는 것입니다.

    12 - MLFQ의 첫 번째 기본 규칙은 우선순위가 더 높은 스레드가 실행된다는 것입니다.

    13 - 두 번째 규칙은 둘 이상의 스레드가 우선순위가 같다면 라운드로빈 방식으로 실행된다는 것입니다.

    15 - 타음 슬라이스를 모두 소진하기 전에 CPU를 양도하면 우선순위는 낮아지지 않습니다.

    14 - 세 번째 규칙과 네 번째 규칙은 작업이 시스템에 진입하면 가장 높은 우선 순위를 갖게 된다는 것과 주어진 타임 슬라이스를 모두 사용하면 우선순위가 낮아진다는 것입니다. 이러한 특성으로 인해 공격에 취약할 수 있습니다. 어떠한 사용자가 스케쥴러를 속여서 지정된 시간보다 더 많은 시간을 할당하도록 하게 할 수 있기 때문입니다.

    16 - 마지막 규칙으로 일정 주기가 지나면 모든 작업의 우선순위를 최상위로 높혀준다는 점이 있습니다.

    17 - 핀토스에서 처음 제공하는 스케쥴링 정책은 라운드로빈입니다. 이번 주의 과제를 해결하며 스케쥴링 방식을 우선순위를 고려한 스케쥴링으로 개선해보았고, mlfq에도 도전해보았습니다.

    18 - 스케쥴링에 만병통치약은 없지만, 작업의 특성을 고려하여 적절한 스케쥴링을 처방해줍시다.

    26 Concurrency and Threads #

    • 스레드와 프로세스의 차이점 쓰레드들은 주소 공간을 공유하기 때문에 동일한 값에 접근할 수 있음
    • 쓰레드 내부
      • pc: 어디서 명령어들을 불러 들일지 추적
      • 레지스터: 연산 수행
    • 문맥 교환 (context switching)
      • 두 개의 스레드가 하나의 프로세서에서 실행 중이라면 실행하고자 하는 스레드는 반드시 문맥 교환을 통해서 실행 중인 스레드와 교체되어야 함
      • T1이 사용하던 레지스터들을 저장하고 T2가 사용하던 레지스터의 내용으로 복원함 → 프로세스의 문맥 교환과 유사
      • 쓰레드 제어 블럭: 프로세스의 쓰레드들의 상태를 저장하기 위해 필요
      • 쓰레드 간의 문맥 교환에서는 주소 공간을 그대로 사용함 (사용하던 페이지 테이블을 그대로 사용)
    • 스택
      • 멀티 쓰레드 프로세스의 경우에는 주소 공간에 하나의 스택이 아니라 쓰레드마다 스택이 할당되어 있음
      • 스택에서 할당되는 변수들이나 매개변수, 리턴 값, 그리고 그 외에 스택에 넣는 것들은 해당 쓰레드의 스택인 쓰레드-로컬 저장소 (thread-local storage)라 불리는 곳에 저장됨

    예제: 쓰레드 생성 # 

    #include <stdio.h>
#include <assert.h>
#include <pthread.h>

void *mythread(void *arg)
{
    printf("%s\n", (char *)arg);
    return NULL;
}

int main(int argc, char *argv[])
{
    pthread_t p1, p2;
    int rc;
    printf("main: begin\n");

    rc = pthread_create(&p1, NULL, mythread, "A");
    assert(rc == 0); // check if the statement is True. if not, exits immediately.
    rc = pthread_create(&p2, NULL, mythread, "B");
    assert(rc == 0);

    // merge waiting threads to make them able to terminate
    rc = pthread_join(p1, NULL);
    assert(rc ==0);
    rc = pthread_join(p2, NULL);
    assert(rc == 0);
    printf("main: end\n");
    return 0;

}

    → 실행 순서는 스케줄러의 동작에 따라 바뀔 수 있다. 먼저 생성되었다고 해서 먼저 실행되는 것도 아님.

    데이터의 공유 #

    • volatile 변수

      volatile의 의미: #

      1. 최적화 방지: 컴파일러는 일반적으로 변수가 변경되지 않을 것으로 예상되면 메모리에서 읽는 대신 레지스터에 저장해 사용하면서 최적화합니다. 그러나 volatile 키워드를 사용하면 변수 값이 외부에 의해 변경될 수 있다고 간주되어 매번 메모리에서 읽습니다.
      2. 다른 스레드 또는 하드웨어에 의해 값이 변경될 수 있는 경우: 주로 멀티스레딩 환경이나 메모리 매핑된 하드웨어 레지스터와 같은 시스템에서, 변수의 값을 자주 참조해야 할 때 사용됩니다.
    • 쓰레드가 공유 데이터를 접근하기 위해 상호작용하는 과정
    #include <stdio.h>
#include <pthread.h>
#include "common.h"
#include "common_threads.h"

static volatile int counter = 0;

void *mythread(void *arg)
{
    printf("%s: begin\n", (char *) arg);
    int i;
    for (i=0; i<1e7; i++) {
        counter = counter + 1;
    }
    printf("%s: done\n", (char *) arg);
    return NULL;
}

int main(int argc, char *argv[])
{
    pthread_t p1, p2;
    printf("main: begin (counter = %d)\n", counter);

    pthread_create(&p1, NULL, mythread, "A");
    pthread_create(&p2, NULL, mythread, "B");

    // merge waiting threads to make them able to terminate
    pthread_join(p1, NULL);
    pthread_join(p2, NULL);
    printf("main: done with both (counter = %d)\n", counter);
    return 0;

}

      → 매 실행의 결과가 다름. 기대한 20000000의 결과가 나오지 않을 때도 있음 .

    제어 없는 스케줄링 (위와 같은 현상이 발생하는 이유) #

    • counter 갱신을 위해서 컴파일러가 생성한 코드의 실행 순서

      1. mov: 명시한 메모리 주소의 값을 읽어들인 후 eax 레지스터에 넣음
      2. add: 1을 eax 레지스터의 값에 더함
      3. mov: eax에 저장되어 있는 값을 메모리의 원래 주소에 다시 저장

    • 쓰레드 1이 counter를 증가시키는 코드 영역에 진입하여 counter의 값을 1 증가시키는 상황 2번까지 진행한 이후에 타이머 인터럽트가 발생 → 운영체제가 실행 중인 쓰레드의 pc값과 eax를 포함하는 레지스터들 등의 현재 상태를 쓰레드의 **TCB(Thread Control Block)**에 저장 쓰레드 2가 선택되고 counter 값을 증가시키는 똑같은 코드 영역에 진입함

      • 쓰레드는 개별적으로 쓰레드 전용 레지스터를 가지고 있다. 사용 중이던 레지스터들을 저장하고 복구하는 문맥 교환 코드에 의해 이 레지스터들은 가상화됨 최종적으로 또 한 번의 문맥 교환이 발생하여 쓰레드 1이 리턴하여 실행되면 레지스터에 있던 값 51이 counter로 저장되어 다시 51이 됨 → 요약하면 counter의 값을 증가시키는 코드가 두 번 수행이 되었지만 50에서 시작한 counter 값은 51

    원자성에 대한 바람 #

    • 경쟁 조건: 멀티 쓰레드가 거의 동시에 임계 영역을 실행하려고 할 때 발생. 명령어의 실행 순서에 따라 결과가 달라지는 상황
    • 비결정적인 결과: 실행할 때 마다 결과가 다른 경우
    • 임계 영역: 공유 자원을 접근하는 코드의 일부분
    • 상호 배제: 하나의 쓰레드가 임계 영역 내의 코드를 실행중일 때는 다른 쓰레드가 실행할 수 없도록 보장해줌
    • 핵심 질문: 동기화를 지원하는 방법
      • 유용한 동기화 함수를 만들기 위해 어떤 하드웨어 지원이 필요한가?
      • 운영체제는 어떤 지원을 해야하는가?
      • 어떻게 하면 이런 함수를 정확하고 효율적으로 만들 수 있을까?
      • 프로그램들이 이 함수들을 활용하여 의도한 결과를 얻으려면 어떻게 해야할까?
    • 이번 장의 목표: 병행 실행이라는 어려운 상황에서 하드웨어 동기화 명령어와 운영체제의 지원을 통해 한 번에 하나의 쓰레드만 임계 영역에서 실행하도록 구성된, “제대로 잘 작동하는” 멀티 쓰레드 프로그램을 작성하기

    상대 기다리기 #

    하나의 쓰레드가 다른 쓰레드가 어떤 동작을 끝낼 때까지 대기해야하는 상황이 빈번하게 발생함

    ex) 프로세스가 디스크 I/O를 요청하고 응답이 올 때까지 잠든 경우

    → I/O 완료 후 잠들었던 프로세스가 깨어나 이후 작업을 진행함
